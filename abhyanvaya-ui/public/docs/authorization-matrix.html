Authorization matrix (API)

Maps ASP.NET authorization policies to roles, JWT permission claims, and extra data rules in controllers (from Program.cs and controller attributes).

Spreadsheet exports: authorization_policies.csv · permission_keys.csv · jwt_permission_resolution.csv · data_layer_rules.csv · authorization_endpoints.csv

How policies combine

  • Class + method [Authorize]: user must satisfy all applicable policies (logical AND).
  • TenantScopedUser: HasTenantHandler — SuperAdmin always passes; others need TenantId claim > 0.

JWT permission resolution (JwtService.ResolvePermissionKeysAsync):

  • If the user has any UserApplicationRole rows → JWT permission claims come only from those roles (no merge with PermissionKeys.All / LegacyFacultySet).
  • If none → Admin gets PermissionKeys.All; Faculty gets PermissionKeys.LegacyFacultySet; SuperAdmin gets every permission row in the database.

2. Permission keys catalog (PermissionKeys)

Stable strings include: Students.View, Students.Manage, Attendance.View, Attendance.Manage, Reports.View, Dashboard.View, Master.View, Setup.Subjects.Manage, Setup.Departments.Manage, Setup.Staff.Manage, Setup.Lookups.Manage, Setup.Courses.Manage, Setup.Groups.Manage, Setup.Semesters.Manage, Organization.Manage.

Many API policies do not assert these strings; they assert role + tenant instead. Keys still matter for setup-style policies and front-end toggles.

3. Data-layer rules (not policies)

Applied after policy checks pass:

Area Rule
Faculty + linked staff (StaffId > 0) FacultySubjectAccess / StaffSubjectAssignment — subject-scoped operations only for assigned subjects.
Faculty, legacy (no staff link) Many endpoints restrict by JWT CourseId / GroupId (cohort).
Reports (/api/reports/student, /api/reports/monthly) Legacy Faculty: 403 if student not same course/group. Staff-linked: attendance filtered to assigned subject IDs + tenant.
Dashboard overview (staff-linked Faculty) Metrics scoped to assigned subjects / related students.

5. Maintenance tips

  1. New endpoints: set policy on the action (or document inherited auth).
  2. New permission key: wire in Program.cs if the API should enforce it.
  3. Application roles replace default JWT permission sets entirely when any assignment exists — inform tenant admins in the RBAC UI.

Reference only — update when policies or controllers change.
